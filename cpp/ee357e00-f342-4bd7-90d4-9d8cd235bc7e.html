Reactor 模型

参考：https://www.jianshu.com/p/588f0a71a9f1

Reactor 模式是一种常用的事件驱动编程模型，用于处理高并发网络应用程序。它基于一个中心“反应器”（Reactor）来监听和接收所有的 I/O 事件，并将这些事件分配给相应的处理程序进行处理。在这个模式中，事件的产生和处理是分离的，使得应用程序可以更加高效地处理多个连接和请求。

Reactor 是一种事件驱动框架，该框架下改变了 select、poll 和 epoll 对 fd 进行管理的思路转变成对读写事件进行管理，重点变成了事件。Reator 逆置了事件处理流程，应用程序需要提供相应的接口并注册到 Reactor 上，在相应的时间发生，Reator 将主动调用这些接口函数，即回调函数

Reactor 模式一般包含以下几个组件：

  • 事件源：描述符由操作系统提供，用于识别每一个事件，如 Socket 描述符、文 件描述符等。在 Linux 中，它用一个整数来表示。事件可以来自外部，如来自客户端 的连接请求、数据等。事件也可以来自内部，如定时器事件。Linux 上是一个整数的描述符，Windows 上就是 Socket 或者 Handle，程序在指定的句柄上注册关心的事件。
  • Event Demultiplexer(事件多路分发机制)：负责管理所有的I/O事件，并将其传递给Reactor进行处理。
    是一个函数，用来等待一个或多个事件的发生。调用者会被阻塞，直到分离器分离的描述符集上有事件发生。Linux 的 select 函数是一个经常被使用的分离器。程序首先将其关心的句柄及事件注册到 event demultiplexer 上，当有事件到达时，event demultiplexer 会发出通知“在已经注册的句柄集中，一个或多个句柄的事件已经就绪”，程序收到通知后，就可以在非阻塞的情况下对事件进行处理了。
  • Reactor(反应堆)：负责监听所有的I/O事件，并将这些事件分配给相应的处理程序进行处理。是事件管理的接口，内部使用event demultiplexer注册、注销事件；并运行事件循环，当有事件进入“就绪”状态时，调用注册事件的回调函数处理事件。对应到 libevent 中，就是 event_base 结构体。
  • Event Handler(事件处理程序)：每个事件都有对应的处理程序，用于处理该事件。事件处理程序提供了一组接口，每个接口对应了一种类型的事件，供Reactor在相应的事件发生时调用，执行相应的事件处理。通常它会绑定一个有效的句柄。对应到 libevent 中，就是 event 结构体。

大致流程如下：

  • 启动 Reactor。
  • Reactor开始监听所有的I/O事件。
  • 当一个I/O事件发生时，Event Demultiplexer 会通知 Reactor。
  • Reactor找到对应的Event Handler，将该事件交给它进行处理。
  • Event Handler处理完该事件后，返回结果给Reactor。
  • Reactor将该结果返回给发起该I/O事件的应用程序。
